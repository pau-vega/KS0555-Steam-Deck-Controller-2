Output: `flatpak/docker-build.sh` script + `just docker-flatpak-build` recipe. Running either produces `RobotController-x86_64.flatpak` at repo root.

  <name>Task 1: Create flatpak/docker-build.sh — Docker-based flatpak-builder runner</name>
  <files>flatpak/docker-build.sh</files>
  <action>
Create `flatpak/docker-build.sh` — a Bash script that runs flatpak-builder inside the same Docker container CI uses.

**Script structure:**

1. **Header**: `#!/usr/bin/env bash` with `set -euo pipefail`
2. **Usage check**: Accept exactly 1 argument: path to the .deb file. Print usage if missing.
3. **Deb validation**: Verify the deb file exists and is non-empty. Error + exit if not.
4. **Platform check**: Verify `docker` is available (`command -v docker`). Print Docker version. Error + exit if Docker not found.
5. **Deb copy**: Resolve script directory (`SCRIPT_DIR`), copy the deb to `flatpak/robot-controller.deb` (same pattern as existing `build.sh` lines 103-113). Print confirmation.
6. **Docker run**: Execute flatpak-builder inside Docker:
   ```bash
   docker run --rm \
     --privileged \
     -v "${REPO_ROOT}:/workspace" \
     -w /workspace \
     ghcr.io/flathub-infra/flatpak-github-actions:freedesktop-2408 \
     /bin/bash -c '
       set -euo pipefail
       echo "→ Configuring Flathub remote..."
       flatpak remote-add --if-not-exists flathub https://flathub.org/repo/flathub.flatpakrepo
       echo "→ Installing runtime + SDK (one-time download ~300 MB)..."
       flatpak install --user -y --noninteractive flathub org.freedesktop.Platform//24.08 org.freedesktop.Sdk//24.08
       echo "→ Running flatpak-builder..."
       flatpak-builder --user --install --force-clean --disable-rofiles-fuse build-dir flatpak/com.ks0555.robotcontroller.yaml
       echo "→ Creating Flatpak bundle..."
       mkdir -p repo
       flatpak build-export repo build-dir
       flatpak build-bundle repo RobotController-x86_64.flatpak com.ks0555.robotcontroller
       echo "✓ Flatpak bundle created: RobotController-x86_64.flatpak"
     '
   ```
   - `REPO_ROOT` = `git rev-parse --show-toplevel` (or `$(cd "$SCRIPT_DIR/.." && pwd)`)
   - `--privileged` is required because flatpak-builder uses bubblewrap for sandboxing, which needs additional Linux capabilities inside Docker
   - `--disable-rofiles-fuse` avoids rofiles-fuse errors common in Docker environments
7. **Post-build cleanup** (after Docker exits): Remove build artifacts:
   ```bash
   rm -rf "${SCRIPT_DIR}/build-dir" "${SCRIPT_DIR}/repo" "${SCRIPT_DIR}/.flatpak-builder"
   rm -f "${SCRIPT_DIR}/robot-controller.deb"
   ```
   Keep only `RobotController-x86_64.flatpak` at repo root.
8. **Report**: Print success message with install/run commands (same format as existing `build.sh` lines 79-82).

**Edge cases to handle:**
- Docker daemon not running → clear error message
- Container image pull may take time on first run → print "Pulling container image (first run may take a few minutes)..."
- Runtime/SDK already installed (subsequent runs) → the `flatpak install` in the container will be a no-op, making warm runs fast

**Do NOT:**
- Do NOT try to build the .deb inside Docker (that requires cross-compilation — out of scope for this script)
- Do NOT use a Dockerfile — the CI container image is used directly via `docker run`
- Do NOT hardcode version numbers — the bundle name `RobotController-x86_64.flatpak` matches the existing naming convention

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host → Docker container | Host filesystem mounted read-write at /workspace; container runs --privileged |
| Network → Container | Container downloads Flathub runtime (~300 MB) over HTTPS; no secrets transmitted |
| Container → Host | .flatpak bundle written to repo root via volume mount; no other host files touched |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-quick-01 | Elevation of Privilege | `docker run --privileged` | accept | Required by bubblewrap in flatpak-builder; container is ephemeral (`--rm`), image is pinned to official Flathub CI; risk is limited to build-time and no production secrets are present |
| T-quick-02 | Spoofing | Flathub remote HTTPS download | mitigate | Flathub uses HTTPS with TLS; `flatpak remote-add` verifies GPG signatures on runtimes; no additional validation needed |
| T-quick-03 | Tampering | .deb file from host mounted into container | accept | The .deb is user-provided from a trusted build (local `cargo tauri build` or CI artifact); if user provides a malicious .deb, the Flatpak sandbox limits its impact; no PII in build artifacts |
| T-quick-04 | Information Disclosure | Volume mount exposes entire repo to container | accept | The container runs ephemeral, no data exfiltration path; only source code and build artifacts are visible (no .env, no credentials in this repo); acceptable for a local build tool |
</threat_model>
