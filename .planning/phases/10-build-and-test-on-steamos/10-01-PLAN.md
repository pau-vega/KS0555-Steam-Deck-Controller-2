  <action>
    Create `.github/workflows/build.yml` for Tauri AppImage builds. Per D-01, D-02, D-14:

    1. **Trigger:** tag push (`v*`) AND `workflow_dispatch`. NOT on every push.

    2. **Job `build`:**
       - `runs-on: ubuntu-latest`
       - No matrix needed — single platform (SteamOS Linux x86_64 only)

    3. **Steps (in order):**
       ```yaml
       - uses: actions/checkout@v4

       - uses: pnpm/action-setup@v5

       - uses: actions/setup-node@v4
         with:
           node-version-file: .nvmrc
           cache: pnpm

       - uses: dtolnay/rust-toolchain@stable

       - name: install system dependencies
         run: |
           sudo apt-get update
           sudo apt-get install -y libwebkit2gtk-4.1-dev libappindicator3-dev librsvg2-dev patchelf

       - name: install frontend dependencies
         run: pnpm install --frozen-lockfile

       - name: build frontend
         run: pnpm build

       - uses: tauri-apps/tauri-action@v1
         env:
           GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
         with:
           projectPath: apps/frontend/src-tauri
           uploadWorkflowArtifacts: true
       ```
       
    4. **Key design rules:**
       - Use `@v4` for checkout, setup-node (matching conventional GHA versions)
       - tauri-action at `@v1` (stable release channel, NOT dev branch — per research)
       - Omit `tagName` and `releaseName` — build-only mode per D-01
       - `uploadWorkflowArtifacts: true` so AppImage is downloadable from CI page
       - `GITHUB_TOKEN` required by tauri-action even for artifact upload
       - Frontend build (`pnpm build`) runs BEFORE tauri-action so `tauri build` finds the dist already compiled
       - `pnpm build` runs turbo which builds `@ks0555/frontend` with Vite output to `apps/frontend/dist/`
       - Use `dtolnay/rust-toolchain@stable` NOT `actions-rs/toolchain` (deprecated)
       - System deps: `libwebkit2gtk-4.1-dev libappindicator3-dev librsvg2-dev patchelf`
       
    5. **Do NOT include:**
       - Matrix strategies (single platform only)
       - Release creation logic (no `tagName`, `releaseName`, `releaseId`)
       - Cross-compile targets (not needed for Linux x86_64)
       - macOS or Windows runners (target is SteamOS Linux only)
       - Deployment steps (AppImage consumed from CI artifacts manually)
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions CI → Repository | CI pipeline runs with GITHUB_TOKEN scoped to the current repository. Limited to artifact upload. |
| CI workflow trigger | Tag push + manual dispatch only. No push-to-main triggers (build is expensive, reserved for releases). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | build.yml workflow | mitigate | Workflow runs on tag push (v*) and manual dispatch only — not on arbitrary push events. GITHUB_TOKEN auto-generated per-run with least privilege. |
| T-10-02 | Denial of Service | CI resource usage | accept | Build is expensive (~10 min). Trigger limited to tag push + manual dispatch. User is sole maintainer — resource exhaustion risk is negligible. |
| T-10-03 | Tampering | AppImage artifact | accept | Artifact is built from source in CI, not uploaded externally. User downloads from CI and verifies on Steam Deck. Binary integrity checked by Tauri build process. |
</threat_model>
