Create `upgrade-robot-controller.sh` at the repo root. Make it executable (`chmod +x`). Self-contained Bash script requiring only `curl` and `jq` (D-16).

**Script structure:**

```bash
#!/usr/bin/env bash
# upgrade-robot-controller.sh — Install or upgrade Robot Controller Flatpak
# Usage: upgrade-robot-controller.sh [--check] [--force] [--help] [--version]
#   --check   Check for updates without installing
#   --force   Skip confirmation prompt
#   --help    Show help
#   --version Show script version
set -euo pipefail

REPO="pau-vega/KS0555-Steam-Deck-Controller-2"
APP_ID="com.ks0555.robotcontroller"
TEMP_DIR="/tmp/robot-controller-upgrade"
API_URL="https://api.github.com/repos/${REPO}/releases/latest"
SCRIPT_VERSION="1.0.0"
```

**Functions to implement:**

1. `check_dependencies()` — verify `curl` and `jq` exist. If `flatpak` is also missing, error out (can't install without it). If `jq` is missing, suggest `sudo apt-get install jq` or `sudo pacman -S jq`.

2. `get_installed_version()` — run `flatpak info --show-commit ${APP_ID} 2>/dev/null` first, then `flatpak info ${APP_ID} 2>/dev/null | grep -i version | head -1`. If flatpak info fails (app not installed), return "none".

3. `get_latest_version()` — curl the GitHub Releases API (D-19). Parse with `jq`:
   - Extract `tag_name` (e.g., `v0.1.5`)
   - Extract `body` (release notes) 
   - Find the asset matching `RobotController-*-x86_64.flatpak` and get its `browser_download_url`
   - Find the asset matching `RobotController-*-x86_64.flatpak.sha256` and get its `browser_download_url`
   - Handle API errors gracefully (rate limiting, network failures)

4. `compare_versions()` — strip `v` prefix from both installed and latest. If installed is "none", return SPECIAL_INSTALL. If versions differ, return UPGRADE_AVAILABLE. If same, return UP_TO_DATE.

5. `download_and_verify()` — 
   - `mkdir -p ${TEMP_DIR}`
   - Download `.flatpak` and `.sha256` to temp dir
   - Run `sha256sum -c RobotController-*.flatpak.sha256` (cd to temp dir first)
   - If verification fails, error and clean up
   - Clean up old `.flatpak` files from temp dir before download (D-18: step 8)

6. `install_flatpak()` — 
   - `flatpak install --user --reinstall ${TEMP_DIR}/RobotController-*-x86_64.flatpak` (D-18: step 7)

7. `print_changelog()` — display the release body from the API.

**Main flow:**
1. Parse flags: `--check`, `--force`, `--help`, `--version`
2. Check dependencies
3. Get installed version and latest version
4. **If fresh install** (no installed version): display "Installing Robot Controller for the first time...", download + verify + install, print success + "Add as Non-Steam Game" instructions
5. **If up to date**: print "✓ Robot Controller is up to date (version X)"
6. **If upgrade available**: print installed vs latest version, print changelog, prompt (unless `--force`): "Install version X? [y/N]". On yes: download + verify + install + clean up
7. **If `--check` flag**: exit after version comparison (D-18: step 3 only)

**Output formatting:**
- Use `set -euo pipefail` for strict error handling
- Use colored output: green for success, yellow for warnings, red for errors
- Use step indicators like `→` for progress, `✓` for success, `✗` for failure
- Include `trap` for cleanup on exit (remove temp dir)

**Edge cases to handle:**
- GitHub API rate limiting: check for `X-RateLimit-Remaining` header, if 0, suggest trying later
- Network failures: retry once with a message, then fail gracefully
- Corrupt download: sha256 verification catches this
- Old `.flatpak` files in temp dir: clean up before downloading new ones
- No `.flatpak` asset found in release: error with release URL for manual check
- Script run without `flatpak` installed: clear error message explaining flatpak is required on Steam Deck

**`--help` output:**
```
Robot Controller — Flatpak Install/Upgrade Launcher

Options:

  --version  Show script version

Examples:
  ./upgrade-robot-controller.sh              # Check and install/upgrade
  ./upgrade-robot-controller.sh --check      # Only check for updates
  ./upgrade-robot-controller.sh --force      # Install without confirmation
```

```just
# --- Flatpak ---

# Build Flatpak bundle from a Tauri .deb
# Usage: just flatpak-build [path-to-deb]
# Default: apps/frontend/src-tauri/target/release/bundle/deb/*.deb
[group('flatpak')]
flatpak-build deb_path="":
    @echo "→ Building Flatpak from deb..."
    @if [ -z "{{deb_path}}" ]; then \
        deb_path="apps/frontend/src-tauri/target/release/bundle/deb/"*.deb; \
    fi; \
    cp "$$deb_path" flatpak/robot-controller.deb
    @echo "→ Running flatpak-builder..."
    flatpak-builder --user --install --force-clean build-dir flatpak/com.ks0555.robotcontroller.yaml
    @echo "→ Creating Flatpak bundle..."
    mkdir -p repo
    flatpak build-export repo build-dir
    flatpak build-bundle repo RobotController-x86_64.flatpak com.ks0555.robotcontroller
    @rm -rf build-dir repo
    @echo ""
    @echo "✓ Flatpak bundle created: RobotController-x86_64.flatpak"
    @echo "  Install: flatpak install --user RobotController-x86_64.flatpak"
    @echo "  Run:     flatpak run com.ks0555.robotcontroller"

# Install a Flatpak bundle
# Usage: just flatpak-install [path-to-flatpak]
# Default: RobotController-x86_64.flatpak
[group('flatpak')]
flatpak-install flatpak_path="RobotController-x86_64.flatpak":
    @echo "→ Installing {{flatpak_path}}..."
    flatpak install --user --reinstall "{{flatpak_path}}"

# Run the installed Flatpak app
[group('flatpak')]
flatpak-run:
    flatpak run com.ks0555.robotcontroller

# Deploy Flatpak bundle to a remote host via scp
# Usage: just flatpak-deploy <hostname> [path-to-flatpak]
# Example: just flatpak-deploy deck@steamdeck.local
[group('flatpak')]
flatpak-deploy hostname flatpak_path="RobotController-x86_64.flatpak":
    @echo "→ Transferring {{flatpak_path}} to {{hostname}}..."
    scp "{{flatpak_path}}" "{{hostname}}:~/"
    @echo ""
    @echo "✓ Flatpak transferred to {{hostname}}:~/"
    @echo "  SSH to the host and run:"
    @echo "    flatpak install --user --reinstall ~/RobotController-x86_64.flatpak"
    @echo "    flatpak run com.ks0555.robotcontroller"
```

| Boundary | Description |
|----------|-------------|
| upgrade-robot-controller.sh → GitHub Releases API | Script fetches release metadata and downloads .flatpak + .sha256 over HTTPS |
| upgrade-robot-controller.sh → flatpak CLI | Script invokes flatpak install with the downloaded bundle |
| justfile → flatpak-builder | just recipe invokes flatpak-builder locally |
| justfile → scp | deploy recipe transfers bundle over SSH |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-08 | Tampering | upgrade-robot-controller.sh download | mitigate | Script downloads `.sha256` alongside `.flatpak`, runs `sha256sum -c` before install. If verification fails, script exits with error and cleans up. |
| T-16-09 | Spoofing | GitHub Releases API response | mitigate | HTTPS enforced for all curl calls. jq parsing validates asset names match `RobotController-*-x86_64.flatpak` pattern before downloading. |
| T-16-10 | Elevation | `flatpak install --user` | accept | `--user` flag installs to user scope (no sudo). This is by design — Flatpak sideload installs are per-user. No privilege escalation risk. |
| T-16-11 | Denial | GitHub API rate limiting | mitigate | Script checks `X-RateLimit-Remaining` header. If 0, prints retry-after time and exits gracefully rather than hanging. |
| T-16-12 | Information Disclosure | scp deployment | accept | `just flatpak-deploy` uses scp which encrypts the transfer. The .flatpak bundle is a public release artifact — no secrets in transit. |
| T-16-13 | Tampering | Temp directory reuse | mitigate | Script uses dedicated `/tmp/robot-controller-upgrade` directory. Old .flatpak files are cleaned before downloading new ones. `trap` removes temp dir on exit. |
</threat_model>
